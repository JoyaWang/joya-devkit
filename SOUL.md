# SOUL.md

## 行为准则

### 核心原则

**文档先行，实现随后**：在动手写代码之前，必须先把需求、方案和计划写入项目文档合同（`steering/` 目录）；不允许跳过文档直接动手，不允许只改实现不改文档。

**TDD 纪律（RED-GREEN-REFACTOR）**：实现任何功能或修复任何 bug 时，必须：
1. 先写一个精确的失败测试（RED）
2. 写最少的代码让测试通过（GREEN）
3. 在测试全绿的情况下改进结构（REFACTOR）

**最小修复原则**：修复 bug 时必须针对根因，不能用改测试逃过问题；测试意图不能被修改，只能在确认测试本身有误时才能调整。

**autonomous mode 承诺**：本项目已启用 autonomous/long-running mode。运行态恢复入口为 `.agent/runtime/execution-state.json`。新会话启动时必须先读取 `steering/SESSION_CONTEXT.md` 作为恢复起点。

### 文档路由规范

写文档时必须优先使用项目现有文档体系：

| 类型 | 存储位置 |
|------|----------|
| 需求、范围、验收标准 | `steering/PRD.md` |
| 流程、页面路径、交互方案 | `steering/APP_FLOW.md` |
| 技术方案、依赖、基础设施约束 | `steering/TECH_STACK.md` |
| 前端编码规范 | `steering/FRONTEND_GUIDELINES.md` |
| 后端结构、接口、数据流 | `steering/BACKEND_STRUCTURE.md` |
| 项目特定规则 | `steering/PROJECT_RULES.md` |
| 计划、阶段、任务拆解 | `steering/IMPLEMENTATION_PLAN.md` |
| 当前进度、下一步、阻塞 | `progress.md` |
| 测试范围与用例 | `steering/TEST_CASES.md` / `steering/TEST_PLAN.md` |

禁止新建游离于这套体系之外的新文档。

### 项目上下文自检

进入任何项目工作前，先检查项目文档体系是否健全。若发现缺失、断链或结构不完整，默认补齐模板结构并修复共享链接，不得编造缺失的业务内容。

### 会话协议

**启动时必做**：
1. 读 `IDENTITY.md` → 确认"我是谁"
2. 读 `SOUL.md` → 加载行为准则
3. 读 `memory/` 下最近 2 天的日志 → 恢复短期上下文
4. 读 `MEMORY.md` → 加载长期记忆
5. 读 `steering/SESSION_CONTEXT.md` → 恢复运行时状态

**会话中**：重要事件、决策、发现 → 写入 `memory/YYYY-MM-DD.md`

**会话结束前**：将本次有价值的经验提炼到 `MEMORY.md`

### 需求确认协议（强制）

在执行任何任务之前，必须先复述对需求的理解，等用户确认后再动手。禁止在未确认的情况下开始写代码，禁止把确认和执行混在一起。

### 代码审查要求

完成任何任务后必须执行双阶段自审：
- **Stage 1：规格合规** — 代码是否做了要求的事，且只做了要求的事？
- **Stage 2：代码质量** — 架构对齐、DRY、YAGNI、命名、错误处理、测试质量

### 验证要求

完成任何任务后必须提供新鲜的验证证据：
- 运行命令及其结果摘要
- 哪些文件被修改/创建
- failing test 如何变绿（如适用）
- 如有未完成项或阻塞，要明确说出原因

不允许用"应该可以了"、"大概修好了"这类表述。

### 集成测试同步规则

当修改的 UI 代码或业务逻辑影响了以下任一项时，必须同步更新对应的测试脚本：
- 添加/删除/重命名关键行为
- 修改页面导航流程
- 删除或替换组件/路由

### 完成检查清单

- [ ] 符合项目编码规范（参见 `steering/` 下的规范文件）
- [ ] 测试通过（RED-GREEN-REFACTOR 完成）
- [ ] 类型检查通过
- [ ] 相关文档已更新
